Q1. What is Software Engineering?

‚úì Correct Answer: Systematic, disciplined approach to software development

Software Engineering is a systematic, disciplined, and quantifiable approach to the development, operation, and maintenance of software. It's not just coding - it encompasses the entire software development lifecycle including planning, design, implementation, testing, deployment, and maintenance.

Q2. What is the first phase of the Software Development Life Cycle (SDLC)?

‚úì Correct Answer: Planning & Requirement Analysis

The SDLC starts with Planning & Requirement Analysis where we gather and analyze what the software needs to do. This phase produces the Software Requirements Specification (SRS) document. You can't design or build software without first understanding what it should do!

Q3. How many phases are there in the traditional SDLC?

‚úì Correct Answer: 6

The traditional SDLC has 6 phases: (1) Planning & Requirement Analysis, (2) System Design, (3) Implementation, (4) Testing, (5) Deployment, and (6) Maintenance. Each phase has specific deliverables and activities.

Q4. Which SDLC model follows a strictly sequential, linear approach where you cannot go back to previous phases?

‚úì Correct Answer: Waterfall Model

The Waterfall Model is characterized by its sequential, linear approach. Once a phase is completed, you cannot go back. This rigidity makes it suitable for projects with well-defined, unchanging requirements but problematic for projects where requirements might evolve.

Q5. Which SDLC model is best suited for a startup building a mobile app with evolving requirements?

‚úì Correct Answer: Agile Model

Agile is perfect for startups because it embraces changing requirements, delivers working software in short iterations (sprints), involves continuous customer feedback, and allows rapid adaptation to market needs. This flexibility is crucial when building innovative products.

Q6. In Agile methodology, how long is a typical sprint?

‚úì Correct Answer: 2-4 weeks

Agile sprints typically last 2-4 weeks. This timeframe is short enough to maintain focus and adapt quickly, but long enough to deliver meaningful, working software. At the end of each sprint, the team delivers a potentially shippable product increment.

Q7. Which SDLC model emphasizes testing at every development phase with corresponding test phases?

‚úì Correct Answer: V-Model

The V-Model (Verification and Validation Model) pairs each development phase with a corresponding testing phase. For example, Requirements phase corresponds to Acceptance Testing, Design corresponds to System Testing, etc. This ensures quality is built-in from the start.

Q8. Which SDLC model is best for large, complex projects with high risk and emphasizes risk analysis?

‚úì Correct Answer: Spiral Model

The Spiral Model combines iterative development with systematic risk analysis. It's ideal for large, complex projects like operating systems or aerospace systems where identifying and mitigating risks early is critical. Each spiral includes planning, risk analysis, engineering, and evaluation phases.

Q9. Which model involves building a working model to understand requirements better before actual development?

‚úì Correct Answer: Prototype Model

The Prototype Model involves creating a quick, working prototype to help stakeholders visualize and understand the system better. This is especially useful when requirements are unclear or for UI/UX-heavy applications. The prototype helps gather better requirements before full development.

Q10. What characterizes a Monolithic Architecture?

‚úì Correct Answer: All components in one codebase, tightly coupled

Monolithic Architecture means all functionality (UI, business logic, data access) is in a single, unified codebase and deployed as one unit. Components are tightly coupled. While simple to develop initially, it becomes difficult to scale and maintain as the application grows.

Q11. Which architecture allows different services to use different technology stacks and scale independently?

‚úì Correct Answer: Microservices Architecture

Microservices Architecture breaks the application into small, independent services. Each service can use its own technology stack (one might use Java, another Python), has its own database, and can be scaled independently. This is used by companies like Netflix, Amazon, and Uber.

Q12. What are the three typical layers in a Layered (3-Tier) Architecture?

‚úì Correct Answer: Presentation, Business Logic, Data Access

The 3-tier Layered Architecture consists of: (1) Presentation Layer (UI/user interaction), (2) Business Logic Layer (application logic and rules), and (3) Data Access Layer (database operations). This separation of concerns makes the system more maintainable and testable.

Q13. Netflix uses which type of architecture to handle millions of users streaming simultaneously?

‚úì Correct Answer: Microservices

Netflix uses Microservices Architecture with hundreds of independent services (user service, recommendation service, streaming service, billing service, etc.). This allows them to scale specific services independently, deploy updates without downtime, and maintain fault isolation.

Q14. What is the main benefit of Layered Architecture?

‚úì Correct Answer: Separation of concerns, each layer has one responsibility

The main benefit of Layered Architecture is separation of concerns. Each layer has a single, well-defined responsibility. This makes the code more maintainable (you can change the UI without touching database code), testable (test business logic independently), and reusable.

Q15. Which SDLC model is the "best" for all projects?

‚úì Correct Answer: It depends on the project requirements

There is NO one-size-fits-all SDLC model! The best model depends on: project size, requirement stability, risk level, timeline, budget, and team expertise. Fixed requirements favor Waterfall, changing requirements favor Agile, high-risk projects favor Spiral, etc.

Q16. What is the main deliverable of the Planning & Requirement Analysis phase?

‚úì Correct Answer: Software Requirements Specification (SRS)

The SRS document is the main deliverable of the Planning & Requirement Analysis phase. It contains detailed descriptions of what the software should do, functional and non-functional requirements, constraints, and acceptance criteria. It serves as a contract between stakeholders and developers.

Q17. Which SDLC phase typically consumes the most time and resources over the software's lifetime?

‚úì Correct Answer: Maintenance

Maintenance typically consumes 60-80% of the total software lifecycle cost! After deployment, software needs continuous bug fixes, security updates, performance improvements, and new features. This phase can last for years or even decades (think of Windows, Java, etc.).

Q18. What does "fault isolation" mean in Microservices Architecture?

‚úì Correct Answer: One service failure doesn't crash the entire system

Fault isolation means if one microservice fails (e.g., payment service), other services (user service, product service) continue working. In a monolithic architecture, one component failure can crash the entire application. This makes microservices more resilient.

Q19. Why is version control (like Git) essential in software engineering?

‚úì Correct Answer: Track changes, collaborate, and rollback to previous versions

Version control is essential because it: (1) tracks who changed what and when, (2) enables multiple developers to work on the same codebase, (3) allows reverting to previous versions if bugs are introduced, (4) supports branching for feature development, and (5) serves as backup. Every professional software team uses version control!

Q20. What is the primary purpose of software testing?

‚úì Correct Answer: Catch bugs, ensure quality, and prevent regressions

Testing is crucial for: (1) catching bugs before users encounter them, (2) ensuring software meets requirements, (3) preventing old bugs from reappearing (regression), (4) building confidence in code changes, and (5) serving as documentation. Fixing bugs in production is 100x more expensive than catching them during development!

Q21. What practice involves team members reviewing each other's code before merging to the main branch?

‚úì Correct Answer: Code review

Code review is a quality assurance practice where team members review each other's code before it's merged. Benefits include: catching bugs early, ensuring coding standards, knowledge sharing, mentorship, and collective code ownership. It's a standard practice in professional software development.

Q22. Why is logging important in software engineering?

‚úì Correct Answer: Debugging, monitoring, audit trail, and performance tracking

Logging is essential for: (1) debugging - identifying where errors occur, (2) monitoring - tracking application health in production, (3) audit trail - knowing who did what and when (required in regulated industries), (4) performance - identifying bottlenecks, and (5) compliance. Professional applications have comprehensive logging at different levels (DEBUG, INFO, WARNING, ERROR, CRITICAL).
